"# PiaMap"

以下の仕様書からデザインの作成をお願いします。

---

PiaMap デザイン依頼仕様書

1. プロダクト概要

PiaMap はストリートピアノを発見・記録・共有するモバイルファーストのSNSです。「近くにあるストリートピアノを見つけて行きたい」「弾いた感想を残して他のピアニストと共有したい」というニーズに応えます。

ドメイン: piamap.com

タグライン: "PiaMap connects people through the love of street piano. Discover, register, share, and celebrate music together."

2. ターゲットユーザー

- 趣味でピアノを弾く人 / アマチュアピアニスト
- 旅先でストリートピアノを探したい人
- ピアノの設置情報を共有したい街の管理者・ファン

ユーザーは大半がスマートフォンからアクセスします。

3. ブランド・トーン

ロゴ・コンセプトデザインがあり、添付の画像 piamap-concept.png を参照してください。

ブランドの軸となる3要素:

- Map Pin (Find Nearby) — 地図ピンによる発見
- Piano Keys (Share Piano) — ピアノを共有する
- Community Together — コミュニティで集まる

ビジュアルの方向性:

- メインのカラーは ブルー〜パープル系のグラデーション (青紫を主調に、暖色アクセントは控えめ)
- モノクロバリエーションも用意されているため、白黒でも成立する設計に
- ロゴは「マップピン + 鍵盤」のシンボル + ワードマーク "PiaMap" の組み合わせ
- 雰囲気: 親しみやすく、敷居が低く、コミュニティ感を尊重する。落ち着いた読みやすさで情報密度は中〜高 (位置情報や属性が多いため)
- アプリアイコンとモノクロ版がコンセプト資料にあるので、UI内のアイコン・シンボル使用はこれに調和させてください

なお現在の実装ではアンバー系を仮で多用していますが、デザインフェーズではブルー〜パープル基調に置き換えてください。

4. ユーザーが達成したいこと

優先度順:

1. 近所や旅先のストリートピアノを地図上で探す
2. ピアノの詳細 (場所・営業時間・楽器の種類・特徴) を確認する
3. 弾きに行った時の感想を投稿する (評価・感想文・写真)
4. 他人の投稿に返信する / いいねする
5. 「行ってみたい / 行ったことある / お気に入り」のピアノをリストとして保存する
6. ピアノ情報 (名前・住所・営業時間など) をコミュニティで編集する
7. 自分や他人のプロフィールから過去の活動を振り返る
8. 名前・特徴・メーカー・評価などで絞り込んで検索する

9. 主要機能と仕様

5.1 認証

- Google ログインのみ (Phase 1)。ゲストでも閲覧は可能、投稿・編集など書き込み系のみ要ログイン
- ゲストが書き込み操作を試みた時は、ログインを促す表示が必要
- ユーザーは @username 形式の custom_id を持つ。3〜20文字、半角英小文字+数字+アンダースコア
  5.2 マップでピアノを探す

- 地図にピアノが点在表示される
- ピン (マーカー) のタップでピアノの簡易情報が出て、そこから詳細へ進める
- 表示中の地図エリアに応じてピアノが取得される
- ピンチイン・アウトで拡大縮小 (UIボタン不要、モバイル前提)
- ピアノが未登録の場所で「ここに新規登録」フローに入れる必要がある
- 既存ピアノを別の位置に動かす編集 (位置修正) も必要

新規登録時の重複候補サジェスト (重要)

ユーザーが新しいピアノを登録しようとした時、選択した位置の 近く (例: 半径100m以内) に既存のピアノがある場合は、

▎ 「もしかしてこのピアノではないですか?」

のような案内とともに該当ピアノを候補として表示します。ユーザーは:

- 候補のピアノをタップ → そのピアノの詳細ページへ遷移 (誤って重複登録するのを防ぐ)
- 候補を無視して新規登録を続行

の2択を選べます。複数候補がある場合は近い順に並べてください。これは MVP の必須機能で、データの重複登録を防ぐ重要なガードレールです。

5.3 ピアノ検索 + フィルタ

検索クエリは ピアノ名のみ に対する部分一致。

絞り込みフィルタ項目:

- ピアノ種別 (グランド / アップライト / 電子 / 不明)
- メーカー (YAMAHA / KAWAI / STEINWAY / BÖSENDORFER / BECHSTEIN / BLÜTHNER / FAZIOLI / BOSTON / DIAPASON / ROLAND / KORG / CASIO / 不明)
- 最低評価 (★3 / ★4 / ★4.5 以上)
- ピアノの特徴 5項目 (それぞれ「N以上」で絞れる):
  - 響き (1=弱い 〜 5=豊か)
  - 調律 (1=悪い 〜 5=良い)
  - 鍵盤の重さ (1=軽い 〜 5=重い)
  - 周囲音 (1=静か 〜 5=賑やか)
  - 人通り (1=少ない 〜 5=多い)

検索体験には以下も含む:

- 入力中のサジェスト (候補リスト)
- 過去に詳細を開いたピアノの履歴表示 (フォーカス時かつ未入力時)
- 候補・履歴をタップしたら、入力欄にピアノ名が自動フィル
  5.4 ピアノ詳細

ピアノごとに以下の情報を表示:

- 表示名 / 説明文
- 住所・都道府県・市区町村
- 会場種別 (駅、商業施設、公園など、自由記述)
- ピアノ種別 / メーカー / モデル / 製造年
- 営業時間 (自由記述)
- 営業状況 (常時 / イベント時 / 要予約 / 休止中など)
- 営業に関する補足
- 設置・撤去日時
- 登録者
- 平均評価 (評価数が0件のときは「未評価」状態を表現)
- 5特徴の集計メーター (count + average を持つ。評価が0の場合は非表示でも可)
- 投稿リスト (ユーザーの感想)
- 編集履歴
- ピアノ情報の編集ボタン (要ログイン)
- ピアノの位置を修正するボタン (要ログイン、地図上のドラッグで移動)
  5.5 投稿 (Post)

「ピアノに行った記録」を残すエンティティ。ユーザーごと・ピアノごとに複数投稿可能。

- 投稿には rating または感想文 (body) のどちらかが必須。両方あっても良い
- rating: 1〜5の星 (任意)
- 感想文: 4000文字以内 (任意)
- 訪問日 (visit_time): 必須
- 5特徴メーター: 各 1〜5 任意 (個別に未評価を許す)
- 公開範囲: public / private (将来 unlisted 追加余地)
- 投稿成立 = 訪問成立、つまり「行ったことある」リストに自動追加される
- 編集・削除は本人のみ
- 削除しても訪問記録は残す

投稿には以下が紐づく:

- いいね数 (他ユーザーから)
- 返信数 (後述)
- 投稿者の表示名・アバター・ピアノ名 (展開フィールド)
  5.6 返信 (Reply)

投稿に対するコメント。

- 本文必須 (2000文字以内)
- 返信の返信 (parent_comment) も技術的には可能 (UIでサポートするかは任意)
- 削除は本人のみ
- 評価などは持たない、純粋に文字でのリアクション
  5.7 いいね

投稿に対するいいね。冪等な追加・削除のみ。集計はピアノ・投稿の両方で持つ。未認証ではタップ時にログイン誘導。

5.8 保存リスト

ユーザーが個人的にピアノを集める3つのリスト:

- 行ってみたい (wishlist): 自発的に追加
- 行ったことある (visited): 投稿時に自動追加 (手動操作も可)
- お気に入り (favorite): 自発的に追加 (=「また弾きたい」)

同じピアノを複数のリストに同時追加可能。リスト一覧と、リスト内のピアノ一覧を表示できる必要があります。他ユーザーのリストも閲覧可。

5.9 プロフィール

自分・他人で表示内容は同等 (一部権限差):

- アバター / 表示名 / @custom_id / 自己紹介
- 統計: 投稿数 / 編集数 / 登録年
- 投稿一覧
- いいねした投稿一覧
- 自分が書いた返信一覧
- 保存リスト一覧 (3つのリストへ遷移)
- 自分のプロフィール時のみ「設定」へのリンク
  5.10 ピアノ情報の編集 (コミュニティ編集)

Wiki的に誰でもピアノ情報を編集できる。

- 編集ログを公開 (誰がいつ何を変えたか)
- 「最終編集表示」「Revert」「編集メモ」をサポート
- 信頼ユーザー (登録から30日経過 + 投稿+編集合計10件以上 + 通報受領0件) のみ:
  - ピアノの完全削除
  - 500m以上の位置移動
  - 名前の全文置換
- 通報閾値超過 (デフォ5件) で自動非公開

編集ログは「フィールド名: 旧値 → 新値」の差分表示が望ましい。新規登録時は「新規登録しました」のメッセージ。

5.11 通報

ユーザー / ピアノ / 投稿 / 返信 などに通報できる。本人または管理者がアクションを取るまで残る。MVP ではユーザー → 管理者通報のみで、UI上は通報ボタン程度。

5.12 通知 (Phase 1 後半)

DB に蓄積、クライアントポーリングで取得 (リアルタイム配信は Phase 2 以降)。種類:

- 自分の投稿への返信
- 自分の投稿へのいいね
- 編集を Watch しているピアノが変更された
  5.13 設定

- アカウント (アバター変更、custom_id 変更、自己紹介)
- ログアウト
- 利用規約・プライバシーポリシーへのリンク
- 開発者用 dev token 入力 (内部利用)

6. ドメインモデル (主要概念)

- User — custom_id で一意。表示名・アバター・自己紹介を持つ
- Piano — 場所・楽器・営業情報を持つ。集計値 (投稿数 / 評価平均 / 5特徴平均 / 各リスト数) を保持
- Post — User × Piano に対する訪問の記録。評価か感想文のいずれか必須
- Reply — Post に対する文字でのリアクション
- Like — Post に対するシンプルな冪等いいね
- List — User × Piano × ListKind (wishlist / visited / favorite) の3種
- PianoEdit — Piano の編集ログ (操作・差分・編集者・要約)
- Notification — 自分宛のイベント蓄積

7. データの状態 (デザインで考慮してほしい)

各リスト・各セクションで以下の状態を意識してください:

- 未認証 (ログイン誘導)
- ローディング中
- 空 (まだ投稿がない / 該当ピアノなし / 履歴なし など)
- エラー (読み込み失敗、ネットワーク不調)
- 成功
- 楽観的更新 (いいね・リスト追加など、即時反映 + ロールバック)
- 部分編集 (元の値を残しつつ一部だけ更新)

8. プライバシー / 安全性

- 画像アップロード時は EXIF 必須削除 (位置情報の漏洩防止)
- 投稿写真は EXIF 読み取って訪問日時の自動入力候補に使うが、保存時はクリア
- アバター画像は EXIF を読まずに削除のみ
- 通報による自動非公開 (5件で hidden 化、admin 解除)
- ユーザー制裁 (凍結・BAN) の可能性 (UIは管理側 = 当面は不要)

9. 著作権 / 利用規約

- 動画はカバー曲多数のため将来的に JASRAC 包括契約を視野
- 投稿物の規約違反は通報削除フローで対応
- 利用規約・プライバシーポリシーは設定からアクセス可能であること
